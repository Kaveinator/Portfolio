The Motive

I've wanted to make a devlog for some time, except I haven't had a clue as to what I should include or how it would look like. For example, I saw some forums contain a catergory for each post alongside with tags. I was already planning to add tags, since it makes searching for content much easier, but didn't understand why I would need another catergory filter. And thought, since I am already able to directly group devlogs with projects (which in turn also connects it to an organization), I could make a filter based on those instead of searching for catergories. Everything else could be searched via a tech stack filter or tag. Therefore, adding yet another searchable term, would just make things confusing to implement and would also hinder the user experience.

Planning

I already have a database for everything related to my portfolio like the contact form submition details, (and recently) projects and organizations and (finally) devlog related tables. Since I already indexed most of my projects into the projects database, I also got the idea of linking projects directly from the devlog post and vice versa, and the most exciting part? This can all be done autonamously, since I made a table dedicated to linking project ids (a number assigned to each project) with devlog post ids (a number assigned to each devlog entry), and adding it to the projects page is a matter of reversing the lookup and then adding it to the project page.
